## ⚠️ 发现项

### 1. 〔高优先 · 建议 9.4 前修〕飞弹击杀掉落触发物理错误 `Can't change this state while flushing queries`
- **现象**：`SHOT_WEAPON=cannon SHOT_SKILL=ground`（飞弹技能）单次连拍报该 ERROR **20 次**。
- **调用链**（引擎给出 backtrace）：
  `projectile._on_body`(碰撞信号, projectile.gd:55) → `enemy.take_damage`(289) → `enemy._die`(310) → `Pickup.spawn`(54) → **`pickup._ready`(76) 在物理 flush 期添加/配置 CollisionShape** → 报错。
- **根因**：飞弹命中是在物理查询 flush 内的信号回调；敌人死亡当场 `add_child` 掉落物，掉落物 `_ready` 立即配置碰撞 shape → 被引擎拒绝（要求 deferred）。**与 9.2 同一类问题，但在新站点（pickup 生成），且是另一句错误信息**。
- **触发条件**：**飞弹击杀且掉落**。纯近战技能(直接 take_damage,非信号)不触发；铁剑地面波若没杀死也不触发（故 sword ground=0）。
- **影响**：刷大量错误日志；掉落物碰撞配置被阻塞，可能首帧拾取判定异常。9.3 飞弹技能(炮全系/全向弹幕)让它高频出现。**很可能 9.3 前就潜伏**（任何飞弹击杀+掉落），只是之前没被发现（见发现项2）。
- **建议修法**：在 `Pickup.spawn` 把 `parent.add_child(p)` 改为 `parent.add_child.call_deferred(p)`；或在 `enemy._die`/`boss._die` 把掉落生成整体 `call_deferred`。改完用全错误扫描复验归零。

### 2. 〔流程 · 重要教训〕错误检测的 grep 太窄，漏报了上面这个 bug
- 9.2 自查、9.3 自查、以及 QA 上一轮 9.3 首检，**都只 grep 了 `"Function blocked during in/out signal"`**，而本 bug 是**另一句** `"Can't change this state while flushing queries"` → 全部漏报（计数=0 的假象）。
- **整改**：QA 连拍验收必须扫描**所有 `ERROR:` 行**，不能只匹配单一字符串。已更新 `.claude/skills/godot-capture/SKILL.md` 的错误检测约定，并修了连拍 harness 的 HUD 武器名刷新（见下）。

### 3. 〔工具修复〕harness 切武器不刷 HUD 武器名（已修）
- `SHOT_WEAPON` 用 `_apply_weapon()` 切武器但不发 `weapon_changed`，导致截图 HUD 一直显示"铁剑"（武器其实已切）。**非游戏 bug**（游戏内 Q 换武器走 `_switch_weapon` 会发信号）。已在 `main.gd:_motion_burst` 补 `weapon_changed.emit`，截图 HUD 现正确。

## 🔧 处理结果（开发线程 · 待 QA 全错误扫描复验）

### 发现项 1〔高优先〕飞弹击杀掉落 `Can't change this state while flushing queries` — 已修
- **修法**：`pickup.gd` 三个生成函数(`spawn`/`spawn_gear`/`spawn_ability`)的 `parent.add_child(p)` 全部改为 `parent.add_child.call_deferred(p)`。这样掉落物的 `_ready`(配置 CollisionShape)推迟到物理 flush 之后，不再在查询 flush 期改物理状态。
- **覆盖面**：在 `Pickup.spawn` 这一层修，**一次覆盖所有掉落来源**——敌人死亡(enemy._die)、Boss 死亡(boss._die)、宝箱开启(chest 的 _on_body 信号里也生成)、房间加载。比只改 enemy._die 更稳。
- 死亡敌的碰撞关闭(`set_deferred("collision_layer"/monitoring)`，9.2 已修)继续保留。

### 复现验证（开发线程 · 全 ERROR 扫描，已吸取发现项2教训）
扫描口径：`... 2>&1 | grep -iE "ERROR" | grep -v "resources still in use|ObjectDB|RID allocations"`（**扫所有 ERROR 行，不再只匹配单句**）：
- `cannon/ground`（原报 20 次）→ **0**。
- **武器(sword/hammer/cannon) × 技能(ground/upper/dash/burst/ult) 共15组** → 总 **0**。
- `hammer/ult` 单独重复 6 次 → **0**；`cannon/ground` 对 jelly/slime/bat 各击杀 → **0**。
- 普通启动 150 帧 → 无 Godot ERROR 行。
- 结论：`Can't change this state while flushing queries` 与 `Function blocked during in/out signal` 两类物理错误均归零。
